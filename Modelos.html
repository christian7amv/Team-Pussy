MODELOS DE CICLOS DE VIDA DEL SOFTWARE

El ciclo de vida del software describe cómo se organiza el proceso de desarrollo, estableciendo el orden de
las tareas, los productos intermedios y los criterios para pasar de una fase a otra.

La elección del modelo es crucial para el éxito del proyecto.

Modelo en Cascada

Es el modelo más antiguo y constituye la base de los demás.

Se caracteriza por ser secuencial, dividiendo el desarrollo en fases que se realizan una detrás de otra:

Características clave

  • Flujo lineal y ordenado, aunque permite retrocesos si se encuentran fallos.
  • Cada fase genera documentación revisada antes de avanzar.

Ventajas

  • Fácil de comprender y planificar.
  • Existe mucho soporte y herramientas asociadas.

Desventajas

  • Es poco realista suponer que el proceso es totalmente lineal.
  • Los cambios se gestionan mal y generan confusión.
  • El cliente no recibe una versión funcional hasta muy tarde.
  • Un fallo detectado tarde implica rehacer fases previas.

Este modelo puede funcionar cuando los requisitos son muy estables y no se esperan cambios.

Modelo de Proceso Incremental

Surge para solucionar un problema del modelo en cascada: el cliente tarda demasiado en
recibir un producto funcional.

Cómo funciona

  • El software se divide en incrementos.
  • En cada incremento se aplican las mismas fases del modelo en cascada, pero en una parte del sistema.
  • Cada incremento entrega funcionalidad parcial, que se amplía en entregas posteriores.

Ventajas

  • Reduce riesgos de retrasos y problemas de aceptación.
  • Los entregables intermedios permiten retroalimentación temprana.
  • El usuario puede validar el sistema a medida que se construye.

Cuándo usarlo

  • Cuando los requisitos no están bien definidos.
  • Cuando es muy probable que se produzcan cambios.

Modelos de Proceso Evolutivo

Estos modelos reconocen que el software evoluciona, los requisitos cambian y es irreal seguir una trayectoria
totalmente lineal.

Incluyen:

Construcción de Prototipos

  • Se crea una versión preliminar del sistema (prototipo) para aclarar requisitos dudosos.
  • No está pensado como producto final: se usa, se evalúa y se descarta o se modifica.
  • Reduce incertidumbre y mejora la comunicación con el cliente.

Modelo en Espiral

Organizado en ciclos que se desarrollan en forma de espiral, cada uno con cuatro actividades:

  1. Planificación: definir objetivos, alternativas y restricciones.
  2. Análisis de riesgos: evaluar alternativas y prevenir problemas.
  3. Ingeniería: desarrollo parcial (análisis, diseño, programación…).
  4. Evaluación: revisión por parte del cliente y decisión de continuar.

Ventaja principal

La gestión explícita y continua del riesgo.

Desventaja principales

Requiere mucha experiencia en análisis de riesgos; si se evalúa mal, el proyecto puede fallar.

Características comunes de los modelos evolutivos (Espiral + Prototipos)

  • El desarrollo es incremental, generando versiones cada vez más completas.
  • Se combinan con el modelo en cascada, pero aplicándolo solo a cada incremento, no al sistema completo.

METODOLOGÍAS RUP Y ÁGILES

El Proceso Unificado de Rational (RUP) es una metodología moderna y estructurada de desarrollo de software
creada por Rational Software Corporation y respaldada por autores como Jacobson, Booch y Rumbaugh (creadores
de UML).

RUP se organiza en dos dimensiones:

1. Dimensión temporal

El ciclo de vida se organiza en ciclos, cada uno compuesto por cuatro fases secuenciales:

Fase de comienzo

  • Se estudia la viabilidad del sistema.
  • Se definen objetivos, alcance, recursos, riesgos, hitos principales y un plan general.
  • Al terminar, se decide si el proyecto continúa.

Fase de elaboración

  • Se analizan los problemas del dominio.
  • Se define una arquitectura base del sistema.
  • Se desarrolla un prototipo arquitectónico validado.
  • Se eliminan los riesgos más importantes.

Fase de construcción

  • Se desarrolla el sistema de forma iterativa e incremental.
  • Se crean versiones cada vez más completas del software.

Fase de transición

  • Se prepara el producto para su entrega al cliente.
  • Incluye ajustes finales y despliegue.

2. Dimensión estática

Define los elementos que participan en el desarrollo:

  • Actividades (activities): Son las tareas a realizar (diseño, análisis, pruebas, etc.).
  • Productos intermedios (artifacts): Documentos, modelos, código, ejecutables, etc.
  • Roles (workers): Los perfiles responsables de actividades.
  • Flujos de trabajo (workflows): Secuencias de actividades. Se dividen en:

A) Flujos de ingeniería

  1. Modelado del negocio
  2. Requisitos
  3. Análisis y diseño
  4. Implementación
  5. Pruebas
  6. Implantación

B) Flujos de apoyo

  1. Gestión de la configuración
  2. Gestión del proyecto
  3. Entorno

METODOLOGÍAS DE DESARROLLO ÁGIL

Las metodologías ágiles surgieron en 2001 con el Manifiesto Ágil, que promueve una forma más flexible y colaborativa de desarrollar software.

Principios clave del desarrollo ágil:

Enfoque en personas y comunicación

  • Los individuos y sus interacciones se valoran más que los procesos y herramientas.
  • La comunicación directa y continua es fundamental.

Software funcional sobre documentación

  • Se prioriza la entrega de software que funcione, reduciendo burocracia y documentación excesiva.

Colaboración estrecha con el cliente

  • El cliente participa continuamente.
  • Se revisa cada entrega y se proporciona retroalimentación frecuente.

Entregas rápidas e incrementales

  • Se entregan versiones funcionales cada pocas semanas.
  • Esto facilita ajustes tempranos y reduce errores graves.

Respuesta rápida al cambio

  • Los cambios son bienvenidos incluso en etapas avanzadas.
  • El coste de incorporar cambios disminuye gracias al desarrollo incremental, pruebas continuas y la retroalimentación constante.

Equipos pequeños y motivados

  • Se fomenta la autonomía del equipo.
  • Se crean entornos de trabajo favorables y colaborativos.

Ventajas destacadas

  • Reducción del coste de cambios.
  • Rápida detección de errores gracias a iteraciones cortas.
  • Mayor satisfacción del cliente.
  • Menor documentación y mayor foco en resultados.
