Modelos de Ciclo de Vida del Software

El ciclo de vida del software describe cómo se organiza el proceso de desarrollo, estableciendo el orden de las tareas, los productos intermedios y los criterios para pasar de una fase a otra. La elección del modelo es crucial para el éxito del proyecto.

1.8.1. Modelo en Cascada (Ciclo de vida clásico)

Es el modelo más antiguo y constituye la base de los demás. Se caracteriza por ser secuencial, dividiendo el desarrollo en fases que se realizan una detrás de otra:

  1. Análisis
  2. Diseño
  3. Programación
  4. Pruebas
  5. Explotación
  6. Mantenimiento

Características clave

Flujo lineal y ordenado, aunque permite retrocesos si se encuentran fallos.

Cada fase genera documentación revisada antes de avanzar.

Ventajas

  • Fácil de comprender y planificar.
  • Existe mucho soporte y herramientas asociadas.

Desventajas

  • Es poco realista suponer que el proceso es totalmente lineal.
  • Los cambios se gestionan mal y generan confusión.
  • El cliente no recibe una versión funcional hasta muy tarde.
  • Un fallo detectado tarde implica rehacer fases previas.

Este modelo puede funcionar cuando los requisitos son muy estables y no se esperan cambios.

1.8.2. Modelo de Proceso Incremental

Surge para solucionar un problema del modelo en cascada: el cliente tarda demasiado en recibir un producto funcional.

Cómo funciona

El software se divide en incrementos.

En cada incremento se aplican las mismas fases del modelo en cascada, pero en una parte del sistema.

Cada incremento entrega funcionalidad parcial, que se amplía en entregas posteriores.

Ventajas (según Cuevas et al. 2003)

  • Reduce riesgos de retrasos y problemas de aceptación.
  • Los entregables intermedios permiten retroalimentación temprana.
  • El usuario puede validar el sistema a medida que se construye.

Cuándo usarlo

  • Cuando los requisitos no están bien definidos.
  • Cuando es muy probable que se produzcan cambios.

1.8.3. Modelos de Proceso Evolutivo

Estos modelos reconocen que el software evoluciona, los requisitos cambian y es irreal seguir una trayectoria totalmente lineal.

Incluyen:

A: Construcción de Prototipos

Se crea una versión preliminar del sistema (prototipo) para aclarar requisitos dudosos.

No está pensado como producto final: se usa, se evalúa y se descarta o se modifica.

Reduce incertidumbre y mejora la comunicación con el cliente.

B: Modelo en Espiral (de Boehm)

Organizado en ciclos que se desarrollan en forma de espiral, cada uno con cuatro actividades:

  1. Planificación: definir objetivos, alternativas y restricciones.
  2. Análisis de riesgos: evaluar alternativas y prevenir problemas.
  3. Ingeniería: desarrollo parcial (análisis, diseño, programación…).
  4. Evaluación: revisión por parte del cliente y decisión de continuar.

Ventaja principal

La gestión explícita y continua del riesgo.

Desventaja principal (según Pressman)

Requiere mucha experiencia en análisis de riesgos; si se evalúa mal, el proyecto puede fallar.

Características comunes de los modelos evolutivos (Espiral + Prototipos)

  • El desarrollo es incremental, generando versiones cada vez más completas.
  • Se combinan con el modelo en cascada, pero aplicándolo solo a cada incremento, no al sistema completo.

Metodologías RUP y Ágiles

Las metodologías de desarrollo de software describen cómo se organiza y gestiona el proceso de creación de un sistema. Entre las más representativas se encuentran RUP y las metodologías ágiles.

RUP (Rational Unified Process)

RUP es una metodología estructurada que organiza el desarrollo en fases y flujos de trabajo. Se basa en el uso de UML y en un proceso iterativo e incremental.

Fases de RUP

1. Inicio (Inception)

Su objetivo es evaluar la viabilidad del proyecto y definir sus objetivos, alcance, restricciones y recursos. Al finalizar esta fase, se decide si el proyecto continúa.

2. Elaboración (Elaboration)

Se analizan los requisitos y se define una arquitectura base del sistema. También se identifican y reducen los riesgos principales.

3. Construcción (Construction)

Se desarrolla y construye el sistema de forma iterativa, generando versiones cada vez más completas.

4. Transición (Transition)

Se prepara y entrega el producto al cliente. Incluye pruebas finales, ajustes y despliegue.

Flujos de trabajo en RUP

Flujos de ingeniería

  • Modelado del negocio
  • Requisitos
  • Análisis y diseño
  • Implementación
  • Pruebas
  • Implantación

Flujos de apoyo

  • Gestión de la configuración
  • Gestión del proyecto
  • Entorno

RUP destaca por su estructura clara, el uso de iteraciones y la importancia que otorga a la arquitectura y la gestión de riesgos.

Metodologías Ágiles

Las metodologías ágiles se basan en un enfoque flexible e iterativo, orientado a la colaboración con el cliente y a la entrega continua de software funcional.

Principios fundamentales

  • Las personas y la comunicación son más importantes que los procesos y herramientas.
  • Se prioriza el software que funciona por encima de la documentación extensa.
  • El cliente participa activamente durante todo el proceso.
  • Se realizan entregas frecuentes de versiones funcionales del producto.
  • La adaptación al cambio es más importante que seguir un plan rígido.
  • Los equipos son pequeños, motivados y autoorganizados.

Las metodologías ágiles permiten reducir el coste de introducir cambios, acelerar la detección de errores y aumentar la satisfacción del cliente debido a la retroalimentación continua.

Conclusión

RUP ofrece una estructura sólida y bien definida basada en fases y flujos de trabajo, mientras que las metodologías ágiles se centran en la flexibilidad, la colaboración continua y las entregas rápidas. Ambos enfoques buscan mejorar la calidad del software, pero se aplican mejor en contextos diferentes según las necesidades del proyecto.
